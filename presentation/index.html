Code generation gets a bad rap. Why?

  • Clunky to work with
  • Hard to update
  • Generates terrible code
Why do I love it? A little history.

It was the early 2000s. We'd just conquered Y2K. Working for a financial systems company.

The business model contained state and federal regulations for banking and loans.

In Rational Rose.

There were hundreds of entities.

This model would change frequently as the lawyers would update it.

The easiest solution was to generate the Java from the UML.

pre-Java EE 1.0 days. ish. JSP 0.92. ish.

Enter BOB: Business Object Builder

hundreds of code snippets each in their own file

So I rewrote it. in XSLT.

IT WAS 2003. Stop judging me.

Its name was JOE. The Java Object Emitter.

This was its own version of hell. But better than what came before.

And it was very effective.

We fixed a persistence bug across hundreds of entities with one commit.

( Well, it was CVS so there were commits for each file involved. ¯\_(ツ)_/¯ )

In all, over 1500 Java classes were managed this way.

And I've been in love with codegen ever since.

Some Benefits

  • Multiple language support
  • Adding a new language becomes "trivial."
  • Consistency across target languages, subsystems.
  • Consistent method definitions.
    • Method and parameter naming
    • Method and parameter ordering
    • Documentation
  • Mass bug fixing

Some Dangers

  • Mass bug generation
  • Fragility is a legit risk
  • The process can become complicated if you're not mindful.
  • You might end up with a low beer truck factor.
  • For the polyglots:
    • Onboarding a new language should be carefully considered
    • Differing capabilities between languages might hamper API development
